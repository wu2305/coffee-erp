# Coffee ERP UI / 交互问题记录

本文件记录预览过程中发现的显示异常、层级表达不清和交互不便问题，并给出建议修复方案与验收标准。

## 问题 01：资料维护页“参数目录”子目录层级不清

### 问题描述

在资料维护页面中，顶部一级分类使用了一组标签按钮：`参数目录`、`咖啡豆`、`烘焙方法`、`烘焙品类`、`方案分类`、`冲煮方案`。

当用户进入 `参数目录` 后，其下方的二级切换按钮 `豆种`、`烘焙度`、`处理法` 与上方一级分类按钮采用了几乎相同的视觉样式。当前界面中，用户难以快速判断这三项是 `参数目录` 的子目录，而不是与一级分类同级的另一组主导航。

### 现象影响

- 用户需要阅读文字内容才能理解层级关系，界面缺少直观的视觉提示。
- 一级分类与二级分类同时出现时，信息层级被压平，增加首次使用成本。
- 切换二级分类时，容易误以为是在切换另一组主功能页，降低资料维护页面的可理解性。

### 代码定位

当前两层标签都复用了同一套样式：

- 一级分类：`src/ui/mod.rs` 中资料维护页分类切换，使用 `tab-item` / `tab-item-active`。
- 二级分类：`src/ui/mod.rs` 中 `ParametersPanel` 的 `豆种` / `烘焙度` / `处理法` 切换，同样使用 `tab-item` / `tab-item-active`。
- 公共样式定义：`src/ui/mod.rs` 中 `.tabs`、`.tabs-six`、`.tab-item`、`.tab-item-active`。

### 根因分析

问题并不是缺少二级分类，而是二级分类没有独立的视觉语言：

- 一级分类与二级分类共用相同边框、背景色、字号、圆角和激活态颜色。
- 二级分类没有额外的容器、标题、分组背景或缩进提示。
- 一级分类与二级分类之间的视觉距离不足，导致用户感知为两组并列导航。

### 修复方案

建议把 `参数目录` 下的切换明确设计为“子目录导航”，而不是复用主标签视觉：

1. 为二级分类增加独立样式类，例如：`subtabs`、`subtab-item`、`subtab-item-active`。
2. 在视觉上弱化二级分类的“主导航感”，强化“当前模块内切换”的提示，可采用以下组合：
   - 使用浅底色容器包裹二级分类；
   - 二级分类按钮改为更轻的底色、较小圆角、略小字号；
   - 激活态使用左侧强调条、内嵌高亮或更浅的品牌色，而不是完全复用一级标签激活样式；
   - 与一级分类之间增加更明确的垂直间距。
3. 在二级分类上方增加说明文字，例如 `参数子目录` 或 `参数类型`，帮助用户建立“当前位于参数目录内部”的认知。
4. 若页面空间允许，可让二级分类整体放入 `panel` 内部，使其在结构上从属于 `参数目录` 内容区，而不是悬浮在一级标签后立即出现。

### 推荐落地方式

优先采用以下最小改动方案：

- 保留一级分类现有 `tab-item` 样式不变；
- 为 `ParametersPanel` 单独新增一套二级分类样式；
- 在二级分类前增加一个低干扰说明标签，如 `参数子目录`；
- 将二级分类包裹在一个浅灰或浅蓝背景的容器中，与主标签形成层级区隔。

这样可以在不改动整体信息架构的前提下，快速建立主次层级。

### 验收标准

修复后应满足：

- 用户无需仔细阅读文案，也能一眼分辨“上方是一级分类、下方是参数目录的子分类”。
- `豆种`、`烘焙度`、`处理法` 看起来属于 `参数目录` 内容区，而不是新的主导航。
- 页面在移动端宽度下仍能保持按钮可点按性和层级清晰度。
- 激活态样式在一级分类与二级分类之间有明确区别。

### 实施优先级

- 优先级：高
- 类型：可用性 / 视觉层级
- 建议处理阶段：资料维护页 UI 优化第一批

## 问题 02：资料页不应单独维护“烘焙方法”，应改为批次 AG 色值记录与烘焙度匹配

### 问题描述

当前资料页面把 `烘焙方法` 作为一级分类单独展示，并提供完整的列表与表单维护能力。但从实际业务流程看，用户在咖啡烘焙完成后，更需要记录的是该批次的 `AG 色值（Agtron）`，并将它与参数目录中的烘焙度范围进行对比，以得到对应的烘焙深度，而不是维护一套独立的“烘焙方法”资料。

换句话说，当前界面暴露的是一个偏内部建模的主数据入口，而用户真正需要的是一个发生在“批次完成 / 入库”流程中的判定动作。

### 现象影响

- 资料页出现 `烘焙方法`，会让用户误以为日常需要维护这类基础资料。
- 用户完成烘焙后，无法在批次录入流程中直接填写 `AG 色值`，业务闭环不完整。
- `烘焙度` 已经在参数目录中维护了范围，但当前系统没有把这个范围真正用于批次判定。
- 用户心中的流程是“录批次 -> 填 AG -> 自动匹配烘焙度”，而当前流程是“先维护烘焙方法 -> 再选择烘焙品类”，两者认知模型不一致。

### 代码定位

当前实现中：

- `src/ui/mod.rs` 的资料页一级分类包含 `烘焙方法`。
- `src/ui/mod.rs` 中存在独立的 `RoastMethodsPanel`，用于维护 `烘焙方法列表` 和 `烘焙方法表单`。
- `src/domain/models.rs` 中 `RoastMethod` 是独立实体，`RoastProfile` 通过 `method_id` 依赖它。
- `src/ui/inventory_page.rs` 的入库表单当前只有 `烘焙品类`、`烘焙完成时间`、`批次数量`、`备注`，没有 `AG 色值` 录入字段。
- `src/domain/models.rs` 中 `RoastLevelOption` 已经具备 `agtron_range`、`agtron_min`、`agtron_max` 字段，说明“按 AG 区间匹配烘焙度”的基础数据结构已经存在。

### 根因分析

这不是简单的 UI 多余项，而是当前业务建模偏向“资料维护视角”，没有贴合真实门店操作流程：

- 把 `烘焙方法` 设计成独立主数据，强化了系统内部分类，却没有服务用户的核心动作。
- `烘焙度` 已有区间定义，但缺少“批次记录实际 AG 值并自动归类”的流程闭环。
- 入库页承担了批次生成，但没有承接烘焙完成后的质量记录与烘焙深度确认。

### 修复方案

建议把“烘焙方法”从当前资料维护主流程中降级或移除，转而把 `AG 色值 -> 烘焙度匹配` 设计为批次流程的一部分。

1. **移除或隐藏资料页一级分类中的 `烘焙方法`**
   - 资料维护首页不再把它作为用户显式维护入口。
   - 如果短期内数据结构暂不调整，可先仅隐藏 UI 入口，避免继续暴露给用户。

2. **在批次流程中增加 `AG 色值` 记录**
   - 在 `入库` 页或“批次完成确认”步骤增加 `AG 色值` 输入项。
   - 录入时允许用户填写单个实测值，而不是手工选择烘焙度。

3. **基于参数目录自动匹配烘焙度**
   - 使用 `RoastLevelOption.agtron_min / agtron_max` 与批次 `AG 色值` 做区间匹配。
   - 匹配成功后，自动给该批次或其关联品类标记对应 `烘焙度`。
   - 如果落在多个区间边界，需定义稳定规则；如果未命中任何区间，应给出“未匹配，请检查参数目录范围”的提示。

4. **把用户心智从“维护烘焙方法”切回“记录批次结果”**
   - 页面语言应围绕 `烘焙结果`、`AG 色值`、`匹配烘焙度` 展开。
   - 用户看到的流程应是测量与确认，而不是先维护一组难以理解的抽象资料项。

### 推荐落地方式

建议分两阶段实施：

- **第一阶段（低风险快速修正）**
  - 从资料页隐藏 `烘焙方法` 一级分类。
  - 在问题说明或产品文档中明确：烘焙深度以后由 `AG 色值` 匹配得到。
  - 保留底层数据结构，避免一次性大改影响现有保存逻辑。

- **第二阶段（流程重构）**
  - 为批次新增 `AG 色值` 字段。
  - 增加 `AG 色值 -> 烘焙度` 自动匹配逻辑。
  - 重新审视 `RoastProfile` 对 `method_id` 的依赖，决定是删除该字段，还是将其改造成更贴近生产工艺的内部元数据，而非用户维护入口。

### 验收标准

修复后应满足：

- 资料页面不再出现让用户困惑的 `烘焙方法` 主入口。
- 用户可以在批次相关流程中记录实际 `AG 色值`。
- 系统能够根据参数目录中的 Agtron 范围自动给出 `烘焙度` 匹配结果。
- 当范围未配置或匹配失败时，界面能给出清晰提示，而不是让用户自行猜测。
- 整体流程更符合“烘焙完成后记录结果”的真实操作顺序。

### 实施优先级

- 优先级：高
- 类型：业务流程 / 信息架构 / 数据建模
- 建议处理阶段：资料维护与批次录入流程联动优化第一批

## 问题 03：从资料页切换到“今日”时未回到页面顶部

### 问题描述

该问题在模拟器中未稳定复现，但在真机环境中可以观察到：当用户在 `资料` 页面，尤其是停留在较长的 `冲煮方案` 内容区域下方时，点击底部导航切换到 `今日`，页面不会回到新页面顶部，而是停留在原先的下部滚动位置，导致 `今日` 页顶部内容没有进入视口。

### 现象影响

- 用户切换到 `今日` 后，第一眼看不到页面标题和顶部卡片，容易误以为页面没有正确切换。
- 真机上长页面切短页面时会出现“落在中下部空白或次要内容区域”的体验问题。
- 底部 Tab 切换缺少“进入新页面”的稳定反馈，破坏主导航的一致性。

### 代码定位

当前顶层页面切换实现在 `src/ui/mod.rs`：

- `App` 组件中使用 `current_page` 控制 `Today / Inventory / Catalog / Settings` 四个主页面切换。
- 底部导航按钮的点击逻辑当前仅执行 `current_page.set(page)`。
- 页面样式中没有单独的滚动容器；当前滚动发生在文档页面本身，因此切换主 Tab 后会沿用浏览器当前的滚动位置。

### 根因分析

这不是 `今日` 页面本身渲染错误，而是主导航切页时没有显式重置滚动位置：

- `资料` 页面中的 `冲煮方案` 区域内容较长，用户通常会滚动到较低位置。
- 切换到底部导航其他 Tab 时，应用只更新渲染内容，没有调用浏览器滚动复位。
- 在真机浏览器环境中，文档滚动位置会被保留，因此新页面会从旧页面的滚动偏移处开始显示。
- 模拟器未复现，通常是因为视口高度、浏览器行为或测试路径与真机不同，但不影响该问题在真实使用中的成立。

### 修复方案

建议把“切换主 Tab 时回到页面顶部”做成统一行为，而不是依赖用户手动上滑：

1. 在主导航切换 `AppPage` 后，显式调用浏览器滚动到顶部。
2. 优先对主页面切换统一处理：`今日`、`入库`、`资料`、`设置` 都保持一致行为。
3. 滚动复位建议在页面状态切换后立即执行，必要时放到一次异步帧后执行，以确保真机浏览器也能稳定生效。
4. 如果后续引入独立滚动容器，则应改为复位该容器的 `scrollTop`，而不是仅处理 `window`。

### 推荐落地方式

优先采用最小改动方案：

- 在 `src/ui/mod.rs` 的底部导航点击逻辑中，除了 `current_page.set(page)` 之外，增加统一的“滚动到顶部”调用。
- 如果直接在点击回调中调用稳定性不足，可把它抽成辅助函数，并在 `current_page` 变化后通过 effect 执行。
- 为避免未来遗漏，建议将该行为封装为主导航切换的默认副作用，而不是分散到单个页面中处理。

### 验收标准

修复后应满足：

- 从 `资料` 页任意滚动位置切换到 `今日`，页面始终从顶部开始显示。
- 从其他主 Tab 互相切换时，也保持统一的顶部复位行为。
- 真机环境下稳定生效，不依赖用户手动回滚。
- 页面标题和顶部主要内容在切页后第一屏即可见。

### 实施优先级

- 优先级：高
- 类型：移动端交互 / 导航体验
- 建议处理阶段：主导航交互优化第一批

## 问题 04：资料页“新增 / 查看 / 编辑”状态不明确

### 问题描述

当前资料页面中的多个模块（如参数目录、咖啡豆、烘焙方法、烘焙品类、方案分类等）普遍采用同一种布局：上方是列表，下方始终展示一个表单；点击列表项上的 `编辑` 后，仅仅是把该条数据静默填充到下方表单中。

这种实现虽然功能上可用，但从用户体验角度看，`新增`、`查看`、`编辑` 三种操作状态没有被清晰地区分，用户很难一眼判断自己当前正在做什么。

### 现象影响

- 用户不知道下方表单当前是“新增一条”，还是“正在编辑某条已有数据”。
- 点击 `编辑` 后缺少明显反馈，容易误以为按钮没有生效，或者不知道刚刚切换了什么状态。
- 列表项本身只展示有限字段，更多时候像“摘要”，却没有真正独立的 `查看` 入口或详情态。
- `保存` / `清空` 这类通用按钮文案语义过弱，不能帮助用户建立当前任务上下文。
- 在移动端长页面中，列表和表单上下分离，编辑目标与编辑区域距离较远，认知负担更重。

### 代码定位

当前这种交互模式在 `src/ui/mod.rs` 中多处重复出现：

- 列表项提供 `编辑` / `归档` 按钮；
- 点击 `编辑` 后，通过设置对应表单的 `editing_id` 与字段值进入编辑态；
- 表单区域始终可见，但标题仍然是固定的 `表单`，没有随着状态变化；
- 主要操作按钮通常只有 `保存` 和 `清空`，没有明确区分“新建保存”和“更新保存”；
- 大多数模块没有单独的 `查看详情` 入口或只读态。

### 根因分析

这不是单个按钮文案问题，而是交互状态设计不完整：

- 系统内部通过 `editing_id` 区分新增或编辑，但这个状态几乎没有被翻译成用户可感知的界面反馈。
- “查看”被隐含在列表摘要中，没有形成独立操作或清晰详情区。
- “新增”也没有被设计为一个明确的入口动作，而是默认通过空表单隐式承载。
- 列表与表单并列显示虽节省开发成本，但在移动端会弱化流程感，让用户难以理解主次和上下文。

### 修复方案

建议把资料页中的操作状态明确建模为用户可感知的三种模式：`查看`、`新增`、`编辑`。

1. **明确新增入口**
   - 在每个资料模块顶部提供显式的 `新增` 按钮，而不是让空白表单默认承担新增含义。
   - 点击 `新增` 后，表单进入清晰的“新建模式”。

2. **明确编辑态反馈**
   - 点击 `编辑` 后，表单标题应切换为类似 `编辑咖啡豆`、`编辑烘焙品类`。
   - 显示当前正在编辑的对象名称或标识，例如 `正在编辑：耶加雪菲 水洗`。
   - 按钮文案改为 `保存修改` / `取消编辑`，避免与新建态混淆。

3. **补足查看态**
   - 列表项可支持点击进入详情卡片、展开区或只读态，而不是只能直接进入编辑。
   - 至少应让用户先“看清当前项”，再决定是否修改。

4. **减少列表与编辑区域的割裂感**
   - 在移动端可考虑采用以下任一模式：
     - 列表页 + 抽屉 / 弹层表单；
     - 列表页 + 展开式详情 / 编辑区；
     - 列表页 + 独立编辑页。
   - 如果短期内继续保留“列表 + 下方表单”，也应在点击 `编辑` 后自动滚动到表单区，并突出显示当前编辑态。

5. **让主操作文案带语义**
   - 新建态使用 `创建` / `新增并保存`；
   - 编辑态使用 `保存修改`；
   - 退出编辑态使用 `取消编辑`；
   - `清空` 仅保留给真正需要重置输入的新增态。

### 推荐落地方式

建议分层推进：

- **第一阶段（低成本 UX 补强）**
  - 给每个表单增加模式标题：`新增...` / `编辑...`；
  - 编辑态展示对象名称；
  - 将按钮文案从 `保存` / `清空` 改为更有语义的版本；
  - 新增 `取消编辑`，并在进入编辑态时自动滚动到表单区域。

- **第二阶段（结构性优化）**
  - 将“查看”与“编辑”拆开；
  - 重新设计移动端下的资料维护流程，减少同屏信息堆叠；
  - 对复杂模块如 `冲煮方案`、`烘焙品类` 采用更清晰的详情 / 编辑分离交互。

### 验收标准

修复后应满足：

- 用户能够明确知道自己当前是处于 `新增`、`查看` 还是 `编辑` 状态。
- 点击 `编辑` 后，界面会出现明显反馈，而不是仅悄悄填充表单。
- 新建与修改的按钮文案、表单标题、取消动作都有明确语义。
- 在移动端场景下，用户不需要来回猜测“当前编辑的是哪一条数据”。
- 对复杂资料项，查看和编辑的路径清晰，不再混为一体。

### 实施优先级

- 优先级：中高
- 类型：UX / 信息层级 / 操作状态反馈
- 建议处理阶段：资料维护交互优化第一批

## 小里程碑：资料页 UX 优化第一批

### 里程碑目标

在不重做整套资料架构的前提下，先修复资料页当前最影响真实使用体验的四个问题，建立更清晰的层级表达、更符合业务的入口结构，以及更稳定的移动端导航与编辑反馈。

### 范围清单

1. **参数目录子层级可视化**
   - 问题：`参数目录` 与其子目录 `豆种 / 烘焙度 / 处理法` 样式过于接近。
   - 解决逻辑：为子目录建立独立视觉语言，让用户一眼看出主次层级。

2. **移除“烘焙方法”主入口，转向 AG 色值匹配流程**
   - 问题：资料页暴露了不符合用户心智的 `烘焙方法` 维护入口。
   - 解决逻辑：先隐藏该入口，再把业务重心迁移到“批次记录 AG 色值并匹配烘焙度”。

3. **主 Tab 切换后自动回到页面顶部**
   - 问题：真机环境中从 `资料` 切到 `今日` 会停留在旧滚动位置。
   - 解决逻辑：将“切换主页面后滚动到顶部”做成统一导航副作用，保证真机和长页面场景稳定。

4. **明确新增 / 查看 / 编辑三种操作状态**
   - 问题：资料页列表与表单长期并列，用户难以判断自己当前处于哪种操作模式。
   - 解决逻辑：增加显式新增入口、编辑态反馈、取消编辑动作，并逐步补足查看态。

### 实施顺序

建议按以下顺序推进：

- **P1**：主 Tab 切换回顶部
- **P1**：参数目录子层级样式区分
- **P1**：资料页表单新增 / 编辑态提示与按钮语义优化
- **P2**：隐藏 `烘焙方法` 入口，并设计 AG 色值匹配的后续重构方案

### 完成标准

本小里程碑完成时，应满足：

- 资料页主次层级更清晰，用户不再混淆主分类与子目录。
- 真机切换底部 Tab 时，页面能稳定回到顶部。
- 资料页的 `新增 / 查看 / 编辑` 状态具备明显反馈。
- `烘焙方法` 不再作为误导性的主入口继续暴露给用户。
- 四个问题均已有明确实现路径，可继续进入具体开发阶段。

### 备注

这是一个面向真实使用体验的“小步快跑”里程碑，优先解决认知成本高、真机影响明显、且改动收益比高的问题；不要求在这一批里一次性完成所有业务重构。
